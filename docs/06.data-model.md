# Data Model - KidCareVax

## Objetivo

O modelo de dados do KidCareVax foi projetado para suportar o acompanhamento da jornada vacinal infantil, permitindo que responsáveis gerenciem múltiplas crianças, acompanhem históricos vacinais individuais e visualizem campanhas de vacinação.

O modelo prioriza simplicidade, facilidade de manutenção e aderência aos requisitos do MVP.

---

# Entidades

## Responsible

Representa o responsável autenticado na plataforma.

### Atributos

| Campo     | Tipo   | Descrição                          |
| --------- | ------ | ---------------------------------- |
| id        | string | Identificador único                |
| name      | string | Nome do responsável                |
| email     | string | E-mail utilizado para autenticação |
| createdAt | Date   | Data de criação do cadastro        |

---

## Child

Representa uma criança cadastrada pelo responsável.

### Atributos

| Campo         | Tipo    | Descrição                   |
| ------------- | ------- | --------------------------- |
| id            | string  | Identificador único         |
| responsibleId | string  | Referência ao responsável   |
| name          | string  | Nome da criança             |
| birthDate     | Date    | Data de nascimento          |
| photoUrl      | string? | Foto da criança (opcional)  |
| createdAt     | Date    | Data de criação do cadastro |

---

## VaccineRecord

Representa um registro vacinal pertencente a uma criança.

### Atributos

| Campo           | Tipo          | Descrição                    |
| --------------- | ------------- | ---------------------------- |
| id              | string        | Identificador único          |
| childId         | string        | Referência à criança         |
| vaccineName     | string        | Nome da vacina               |
| scheduledDate   | Date          | Data prevista para aplicação |
| applicationDate | Date?         | Data de aplicação            |
| status          | VaccineStatus | Status da vacina             |

---

## Campaign

Representa campanhas de vacinação disponíveis na plataforma.

### Atributos

| Campo          | Tipo    | Descrição                       |
| -------------- | ------- | ------------------------------- |
| id             | string  | Identificador único             |
| title          | string  | Nome da campanha                |
| description    | string  | Descrição da campanha           |
| targetAgeGroup | string  | Público-alvo                    |
| startDate      | Date    | Data de início                  |
| endDate        | Date    | Data de término                 |
| active         | boolean | Indica se a campanha está ativa |

---

# Enumerações

## VaccineStatus

Define o estado de um registro vacinal.

| Valor   | Descrição       |
| ------- | --------------- |
| APPLIED | Vacina aplicada |
| PENDING | Vacina pendente |
| OVERDUE | Vacina atrasada |

---

# Relacionamentos

## Responsible → Child

Um responsável pode possuir uma ou mais crianças cadastradas.

```text
Responsible
    │
    └───< Child
```

Cardinalidade:

```text
1 : N
```

---

## Child → VaccineRecord

Uma criança pode possuir múltiplos registros vacinais.

```text
Child
    │
    └───< VaccineRecord
```

Cardinalidade:

```text
1 : N
```

---

## Campaign

Campanhas são entidades independentes e podem ser consultadas por qualquer responsável autenticado.

```text
Campaign
```

# Decisões de Modelagem

## DM-01

Foi adotada uma entidade Responsible para representar o responsável autenticado na plataforma.

---

## DM-02

Cada criança possui histórico vacinal próprio e independente.

---

## DM-03

A idade da criança não será armazenada.

Ela será calculada dinamicamente a partir da data de nascimento, evitando redundância de dados.

---

## DM-04

As informações da vacina foram incorporadas diretamente ao registro vacinal (VaccineRecord), evitando a criação de uma entidade Vaccine separada para o MVP.

Essa decisão reduz a complexidade da implementação sem comprometer os requisitos do desafio.

---

## DM-05

O armazenamento foi projetado considerando o uso do Firebase Firestore, privilegiando simplicidade e rápida evolução da aplicação.
